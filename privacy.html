Privacy Policy

Last updated: June 4, 2026

Hearth Cmd ("we", "our", "the Service") is operated by David Farrell. This policy describes what data we collect, how we use it, and your rights regarding that data.

1. Data We Collect

Hearth Cmd collects the data necessary to authenticate you, route permission requests to your device, and let you organize AI agents across hosts:

  • Email address: You sign in by entering your email address; we send a 6-digit one-time passcode (OTP) to that address. Your verified email is stored on the server and used to identify you across devices, to deliver invitations to organizations you've been added to, and to contact you about your account. If an organization admin invites someone, the invitee's email address is stored on the server until the invite is accepted, revoked, or expires.
  • Account identifiers: When you first verify your email, the server creates a user record with a randomly generated UUID. Additional UUIDs identify the organizations you belong to, the AI agents and hosts you create, and the devices you sign in from. These identifiers are not linked to your Apple ID.
  • Display name (optional): If you set a name on your user profile, it is stored on the server so other members of your organizations can see who they are collaborating with.
  • Device identifier & secret: A randomly generated UUID and cryptographic secret are created on your device on first sign-in and stored in your iCloud Keychain. They authenticate the app to our relay server. The secret is stored hashed on the server.
  • Push notification token: A token issued by Apple Push Notification service (APNs), used solely to deliver push notifications when the app is not in the foreground. The token is device-specific and does not identify you personally.
  • Organization, host, and agent metadata: When you create an organization, a host registration, an AI agent, a job description, or a working directory, the labels and configuration you provide are stored on the server. This includes:
    • Organization names you choose (e.g. "Smith Household").
    • Hostnames of the machines where the daemon runs.
    • Filesystem paths you designate as agent working directories (e.g. /Users/alice/projects/website).
    • Display names you give to AI agents (e.g. "Alice's Gardener") and to your devices (e.g. "Alice's iPhone").
    • The job-description text you write to direct an agent — title, mandate, and required skills — which is free-form Markdown that you author.
    • The list of AI brain models, harnesses (Claude Code, Codex, etc.), and skills configured for each agent.
  • Permission request data: When an AI agent attempts a tool call (a shell command, file read or write, network request, etc.) the request content — tool name, tool input (which may include commands and file paths), and contextual metadata — is relayed through our server to your device. Your decision (Allow / Deny / Always Allow) and the timestamp are logged on the server for operational, audit, and dispute-resolution purposes.
  • Auto-approve rules: If you choose "Always Allow" for a request, a pattern-based rule (e.g. Bash(git *)) is stored on the server, scoped to the position of the agent it was created for. You can view and delete these rules at any time in the app.
  • Subscription data: If you purchase a Hearth Cmd Pro subscription, the App Store transaction identifier, product identifier, environment (sandbox or production), purchase and expiration timestamps, and the signed transaction (a JWS issued by Apple) are stored on the server to verify and maintain your Pro entitlement. We do not receive your payment information from Apple.
  • Activity stream data: While an agent session is open, the agent's transcript entries (tool calls, text responses, user messages) stream through our server to your device in real time. This data is relayed in memory only — it is never written to disk, logged, or stored on our servers. Only the permission-request metadata described above is persisted.
  • OTP request IP address: When you request a one-time passcode, the originating IP address is recorded on the server for abuse rate-limiting on that endpoint. It is not used for tracking, profiling, or analytics.
  • Google OAuth credentials (optional): If you connect a Google account to use the Google Calendar, Google Drive, or Google People plugins, we store an OAuth refresh token on the server, encrypted at rest using a key pinned to your host machine (X25519 + ChaCha20-Poly1305). Only the daemon running on your host can decrypt this token. A short-lived access token is derived from the refresh token on demand and used solely for the API call the agent is attempting; the access token is never stored. We also store the Google account email address associated with the connection as a human-readable label so you can identify which account is connected.

2. Data We Do Not Collect

  • We do not collect your Apple ID or Apple-supplied advertising identifier (IDFA).
  • We do not collect analytics, usage metrics, or product-interaction telemetry beyond what is described above.
  • We do not use cookies or third-party tracking on the app or the marketing site (the site's only third-party script is Google Analytics for aggregate site-traffic stats; the app does not load it).
  • We do not access your source code, files, or any data on your computer beyond what is included in a permission request that the agent itself sends to the server.
  • Activity-stream transcripts are not retained on our servers.
  • We do not use Google user data — including Calendar events, Drive file content, or Contacts — to train machine learning models, serve advertising, or for any purpose other than completing the specific action the agent requested on your behalf.
  • Google user data accessed through the Calendar, Drive, or People APIs is passed directly to the agent running on your own machine. It is not stored on our servers beyond the duration of the individual API call.

3. How We Use Your Data

All data collected is used exclusively to provide the core functionality of the Service:

  • Authenticating you via email OTP and maintaining your sign-in across devices.
  • Routing permission requests from your AI agents to your devices and other members of your organization.
  • Sending push notifications to wake the app when a request arrives or when an agent needs your input.
  • Evaluating auto-approve rules to handle matching requests without prompting you.
  • Delivering and tracking organization invitations.
  • Verifying and maintaining your Hearth Cmd Pro subscription.
  • Performing Google Calendar, Drive, or People API calls on behalf of your agents, using only the access you explicitly granted via the OAuth consent screen.

We do not sell, share, or transfer your data to any third party, except (a) Apple Push Notification service for the sole purpose of delivering notifications, (b) Apple's StoreKit servers for verifying subscription transactions, (c) the email service used to send OTPs and invitations, and (d) Google's APIs when completing an action your agent requested using a Google integration you connected.

Our use of Google API Services and any data received through those services adheres to the Google API Services User Data Policy, including the Limited Use requirements.

4. Data Storage and Security

  • Your device identifier and secret are stored in your iCloud Keychain, which is end-to-end encrypted by Apple.
  • The relay server stores your account record, organization data, agent and host metadata, hashed device secrets, push tokens, auto-approve rules, and subscription state in a database hosted on infrastructure we control.
  • One-time passcodes are stored hashed (SHA-256), not in plain text, and expire after a short window.
  • All communication between the app and the server uses TLS encryption (HTTPS/WSS).
  • Permission request metadata (tool name, tool input, decision, and timestamps) is logged for operational, audit, and dispute-resolution purposes. Active log files are rotated automatically; older logs may be moved to cold storage and retained indefinitely.
  • Google OAuth refresh tokens are encrypted at rest using X25519 key agreement and ChaCha20-Poly1305 authenticated encryption, keyed to your host machine's public key. The plaintext token is never stored on disk on our servers and can only be decrypted by the daemon running on your host.

5. Data Retention

Account records, organization and agent metadata, auto-approve rules, and subscription records are retained for as long as you use the Service. Permission-request logs may be archived to cold storage and retained indefinitely for audit and compliance purposes. To delete your account and associated data, contact us at support@hearthcmd.com; we will remove your user record, organization memberships, device registrations, and auto-approve rules. Permission-request log entries may be retained in audit archives in pseudonymous form (referenced by UUID).

6. Children's Privacy

Hearth Cmd is not directed at children under 13 and we do not knowingly collect data from children.

7. Changes to This Policy

8. Contact

If you have questions about this privacy policy or your data, contact us at support@hearthcmd.com.

9. Google API Services

Hearth Cmd offers optional integrations with Google Calendar, Google Drive, and Google People via OAuth 2.0. This section describes how data from those services is handled.

What we access

  • Google Calendar (calendar.events scope): Calendar events in the calendars you have access to, including event titles, times, attendees, descriptions, and locations.
  • Google Drive (drive.file scope): Files created by Hearth Cmd or explicitly opened through Hearth Cmd, including file names, metadata, and content.
  • Google People (contacts.readonly scope): Contact names, email addresses, phone numbers, job titles, and departments from your Google Contacts or Workspace directory.

How we use it

Google user data is accessed solely to complete the specific action your AI agent is attempting — for example, reading your calendar to check availability before scheduling a meeting, or looking up a contact's email address. Each action is either approved by you in real time or governed by an explicit auto-approve rule you created. Data accessed through Google APIs is passed to the agent running on your own machine and is not stored, logged, or retained on our servers beyond the duration of the individual API call.

What we do not do with Google user data

  • We do not store Google Calendar, Drive, or People data on our servers.
  • We do not use Google user data to train or improve machine learning models.
  • We do not use Google user data for advertising or transfer it to advertising networks.
  • We do not share Google user data with third parties except as necessary to complete the action the agent requested (i.e., the Google API call itself).
  • We do not use Google user data for any purpose other than providing the in-app feature the user requested.

OAuth tokens

When you connect a Google account, an OAuth refresh token is stored on our server encrypted to your host machine's public key (X25519 + ChaCha20-Poly1305). Only the daemon running on your host can decrypt it. Short-lived access tokens are derived on demand and used only for the API call in progress. You can disconnect a Google integration at any time from the app; doing so revokes the stored token.

Our use of Google API Services adheres to the Google API Services User Data Policy, including the Limited Use requirements.
